# Zehn Agent State-Cache Plan

Date: 2026-05-16
Status: research + plan document. No code changes.
Author note: written autonomously while Ali is away. Investigation notes captured first; synthesized plan at the end.

## TL;DR

The CEO, COO, and every cron-driven agent currently re-scan 79 LogicIgniter repos with ~200-260 `gh` API calls every cycle. Hard data from the gateway log: **average 720s wall-clock per heartbeat work-selection turn, 48 turns/day → ~9.6 hours/day of LLM compute** just for "is there anything to do" checks. The output is a 4.8 KB JSON summary whose contents barely change between consecutive scans.

The Yaad doctrine already mandates "don't rediscover" — but Yaad isn't the right store for fast-changing `gh` state, so the rule has no teeth for this case.

**Proposal:** a single deterministic refresher writes a shared runtime-state cache every 10 minutes; all consumers read the cache. Three phases:

- **Phase 1 (MVP, no Go code):** workspace-state cache + deterministic refresher + operating-prompt updates. Cuts heartbeat work-selection from 720s → <30s.
- **Phase 2:** Go-side tool replaces file cache for cleaner type-safety.
- **Phase 3:** GitHub webhooks drive near-real-time invalidation.

Phase 1 alone delivers ~95% of the win. Estimated effort: 1-2 days for the MVP. No upstream-clean code modified.

## Problem statement (one sentence)

Every cron-triggered agent turn (heartbeat work-selection, ops monitor, CEO daily sync, CTO ladder sweep) **rediscovers the LogicIgniter repo set, the open-issue surface, the open-PR surface, the label state, and the dirty-repo state from scratch** by shelling out `gh` and `git` against ~88 LI directories per run, when the underlying data changes far more slowly than the scan cadence.

### Observation 1 — The actual scan script li-coo runs every heartbeat

Recovered the inline Python script (`li_coo_heartbeat_<ts>.py`) from a recent gateway log event. Structure:

```python
ROOT = Path('/Users/aliai/logicigniter')

# 1. Walk the LI tree, find every directory whose `.git` is a real dir
for dirpath, dirnames, filenames in os.walk(ROOT):
    if '.git' in dirnames:
        # `git remote get-url origin` to derive canonical owner/repo
        # Skip non-logicigniter remotes
        # `git status --porcelain=v1 -b` to flag dirty repos

# 2. For every discovered repo (79 of them):
for rp in repos:
    # `gh issue list --repo <full> --state open --limit 100 --json ...`  (~30s)
    # `gh pr list --repo <full> --state open --limit 100 --json ...`     (~30s)

# 3. For every claimed or blocked issue:
#    `gh api repos/.../issues/N/comments --jq '.[-5:]'`  (one more API call per item)

# 4. Classify into:
#    relevant / ready / claimable / claimed / blocked /
#    approval / malformed / needs_human / area_no_status

# 5. Write summary.json to /tmp/li_coo_heartbeat_<ts>/summary.json
```

Source: the full ~10,742-char script lives in `agent.tool.exec` events with `caller=registry.go:194`. Generated fresh by the agent each heartbeat — even the *script itself* is regenerated by the LLM rather than being a stored asset.

### Observation 2 — The cost per scan, the cost per day

Hard numbers from `pkg/agent/agent_event.go:145 turn_end` events:

| Metric | Value |
| --- | --- |
| Repos walked per scan | 79 |
| `gh` API calls per scan (issue + PR + comments) | ~200-260 |
| Wall-clock per li-coo turn (recent 10 samples) | **avg 720s (12 min), min 237s, max 1399s (23 min)** |
| Heartbeat firing cadence | every 30 min |
| Heartbeat-work-selection turns per day | ~48 |

| **Daily `gh` API call volume from rescans alone** | **~10,000+ calls** |

The output of all that work — `summary.json` — is **4,766 bytes**, and most fields are 0 because no claimable work exists most of the time.

This is the central paradox of the current model: **the cost of asking "is there anything to do" is enormous, the answer is "no" most of the time, and the answer barely changes between consecutive asks.** The COO scoreboard, ops monitor, and CEO daily sync all incur similar costs by doing their own (slightly different) scans on top.

### Observation 3 — What the operating prompts actually instruct

`.picoclaw/workspace/memory/LOGICIGNITER_WORK_SELECTION.md` (26 KB) is the canonical algorithm. It describes the queue as the union of:

1. open GitHub issues with `zehn:ready` + `area:*` labels;
2. open PR review queue;
3. blocker retry register.

Trigger model documented at the top:

> "Triggered by heartbeat ('any new work?') **or by GitHub webhook events** (issue labeled `zehn:ready`, PR opened, etc.). Not triggered by cron."

The webhook path is named in doctrine but not wired today. Heartbeat is the only live trigger, so every cycle does the full scan.

The doc does NOT specify a cache. It expects fresh `gh` data every cycle.

### Observation 4 — Yaad's role and the gap

`.picoclaw/workspace/memory/LOGICIGNITER_YAAD_SCHEMA_CONTRACT.md` is explicit:

> "**Persistent rediscovery of the same facts every run is a signal that Yaad is being bypassed**; the responsible agent should write the fact back to Yaad on the next opportunity so future runs do not repeat the work."

> "Before scanning the filesystem for LogicIgniter company structure, app/service inventory, prior decisions, ladder/readiness state, blocker history, customer or persona context, or anything previously recorded by another agent, query Yaad first."

**The contract already mandates the right behavior — but the system doesn't honor it for `gh` state.** The doctrine pairs nicely for *slow-changing structural facts* (company org, service inventory, prior decisions) but breaks down for *fast-changing issue/PR state* because:

- Yaad doesn't hold the current `zehn:ready` set;
- Yaad doesn't hold which PRs are awaiting review;
- Yaad doesn't hold which repos are currently dirty;
- Even if it did, Yaad isn't the right store: it's designed for durable decisions, not live state.

So agents have no choice today but to re-scan `gh`. The doctrine's "don't rediscover" rule applies to architecture/ladder/decisions (which agents already follow); it doesn't and can't cover live operational state.

### Observation 5 — Existing state-cache patterns in the codebase (or lack thereof)

Searched for any structured state cache in `.picoclaw/workspace/` or `pkg/`:

- `.picoclaw/workspace/memory/scoreboard/LATEST.md` — COO scoreboard output, daily-refresh; structured Markdown not JSON; not consumed by other agents as state input (only as visible report).
- `.picoclaw/workspace/memory/LADDER_SNAPSHOT_LATEST.md` — ladder classification (yesterday's CTO sweep); Markdown table; static, not refreshed.
- No file at `.picoclaw/workspace/memory/li-state.json` or similar.
- No Go-side cache: grep for `type.*StateCache|type.*WorkSelectionCache|type.*RepoState` returns nothing.
- No tool registered to serve cached state. Every scan is per-agent, ephemeral, regenerated.

### Observation 6 — The output is small, the input is large, state changes slowly

The summary.json schema (top-level keys):

```
repos: array[79]               ← changes maybe once per week (new repo creation)
labels_queried: array[21]      ← constant
repo_count: 79                 ← constant
total_relevant_issues: N       ← changes only when issues land/close
claimable_ready: N             ← changes only when label flips
claimed: N                     ← changes only on claim/release
blocked: N                     ← changes only on label flip
approval_items: array          ← changes only on label flip
prs: array[N]                  ← changes only on open/close/merge
dirty: array[N]                ← changes when an agent fails to clean up
ts: timestamp                  ← obvious
```

In real life: between two consecutive 30-minute heartbeats, **typically 0-2 fields differ**. Often nothing differs (the most common outcome is "no claimable work").

This means a **TTL of 5-15 minutes on the cache is more than sufficient** for most operational decisions. Webhooks (when wired) can drive zero-latency cache invalidation for the cases that matter (new `zehn:ready` issue lands).

### Observation 7 — Comparable patterns in similar systems

In the broader landscape, the standard solution for this exact problem is a **read-through cache with a freshness contract**:

- A single producer (refresher) hits the expensive source on a schedule;
- Many consumers read the cached output with negligible cost;
- Consumers honor a freshness bound (e.g., "this data is at most 10 min old").

Concretely: GitHub Action runners cache `gh` outputs the same way. Kubernetes informers cache cluster state. Build systems cache dependency graphs. Web apps cache database queries with TTL. Every "many readers, slow source" system uses this pattern.

The PicoClaw runtime already has a `tools.cron.exec_timeout_minutes` budget concept and an `eventBus` to publish state changes. It also has Yaad MCP for durable memory. None of these are used as a state cache today.

## Proposal — three phases, additive

A read-through cache with a single producer (refresher) and many consumers (every agent). Three phases, each independently shippable; each later phase strict superset of the prior one in capability.

### Phase 1 — Workspace-file shared cache (MVP, no Go code)

**One deterministic refresher, one runtime state file, and prompt updates so every agent reads it.**

Components:

1. **State file**: `.picoclaw/workspace/state/li_state_cache.json`
   - Same schema as the current `/tmp/li_coo_heartbeat_<ts>/summary.json` (proven schema, 4.8 KB typical).
   - Adds a `cache_meta` block at the top: `{generated_at, expires_at, generator_agent, refresh_count, schema_version}`.
   - Atomic writes: refresher writes to `li_state_cache.json.tmp` then `rename(2)` to final path.
   - Lives under workspace `state/`, not `memory/`, because this is fast-changing operational state, not long-term memory.

2. **Refresher runner**: `li-state-refresher` every 10 minutes.
   - Schedule: `*/10 * * * *`
   - Target: deterministic local execution, not `agent_turn`. This can be a LaunchAgent, a supervised script, or a PicoClaw command cron only after command jobs are explicitly enabled and verified. It must not consume an LLM turn for data collection.
   - Payload: tightly bounded — execute a stored shell script (or a Go tool — see Phase 2) that does the existing scan, writes the JSON, returns a one-line status.
   - The scan script lives as an asset: `operations/refresh-li-state.sh` (new file), so it's not regenerated by the LLM each run.
   - 10-min cadence gives at most 10 min staleness; heartbeat work-selection then completes in seconds, not minutes.

3. **Operating-prompt updates** so consumers READ first:
   - `logicigniter-ceo-daily-sync.md` (added in v3): before any `gh`-based scan, read `li_state_cache.json` and decide forward move from cached data. Fall back to scan ONLY if cache is missing or older than `cache_meta.expires_at + 5 min` grace.
   - `zehn-operations-monitor.md`: same pattern; the monitor checks cache freshness as one of its health metrics.
   - The heartbeat work-selection delegation message (in `jobs.json`): change from "run the inline Python script" to "read `workspace/state/li_state_cache.json` and apply `LOGICIGNITER_WORK_SELECTION.md` filter+rank on cached data."
   - New doc `.picoclaw/workspace/memory/LI_STATE_CACHE_CONTRACT.md` capturing the cache contract: where it lives, how often it refreshes, schema, freshness bounds, fallback rule.

4. **Schema versioning**: `cache_meta.schema_version = 1` so future cache layout changes don't silently break consumers.

5. **Observability hooks**:
   - Refresher writes a one-line status to `.picoclaw/workspace/state/li_state_cache.status` on each run (`ok | partial | failed | <reason>`).
   - COO daily scoreboard reads `cache_meta.generated_at` and flags if cache is >30 min stale (refresher dead).
   - Ops monitor flags >2 consecutive refresher failures as an actionable Zehn-side issue.

**Risk / mitigation**:

- **Stale-cache decision risk**: agent makes a claim based on 9-min-old data; a new ready issue lands in that window and is missed. Mitigation: 10-min TTL + heartbeat still fires every 30 min as backstop. Worst-case latency for a new issue: ~10 min, vs. current ~30 min anyway.
- **Refresher fails silently**: cache goes stale, agents either fall back to scan (slow but correct) or proceed on stale data (could re-claim an issue someone already finished). Mitigation: the fallback path is "scan with the existing script"; nothing breaks structurally.
- **Cache file corruption**: atomic rename + schema_version + on-read JSON validation. Bad file → fall back to scan.

**What this DOESN'T do**:

- Doesn't fix the underlying issue that `gh` API is the only source of truth (Yaad limitation noted above).
- Doesn't reduce *cron* firings; it reduces *per-firing* work. Same heartbeat cadence, but each heartbeat becomes a cache read instead of a 12-min scan.
- Doesn't change the `LOGICIGNITER_WORK_SELECTION.md` algorithm; only its *input source*.

**Estimated impact (Phase 1 only)**:

- Average heartbeat work-selection turn: **720s → <30s** (cache read + LLM decision + Yaad write).
- Daily compute saved: ~9 hours of gateway-occupying work freed for actual work execution.
- Daily `gh` API calls: 10,000+ → ~1,200 (one refresher × 79 repos × ~2 calls × 144 refreshes/day) — and these are bounded, well-paced, less likely to hit rate limits.
- LLM token cost per heartbeat: drops by an order of magnitude (no need to feed raw `gh` JSON into the prompt).

### Phase 2 — Go-side state cache as a built-in tool

After Phase 1 is stable and Phase 1's prompt updates have proven the pattern, replace the file-cache + Python refresher with a Go-side in-memory cache plus a registered tool.

Components:

1. **New file `pkg/agent/zehn_li_state_tool.go`** (Zehn-only, prefixed `zehn_` for upstream-sync safety):
   - Implements a `tools.Tool` named `get_li_state` that returns cached state.
   - Implements `refresh_li_state` for on-demand refresh.
   - Internal cache: in-process map keyed by snapshot timestamp. Refreshes in a goroutine.
   - On cache miss: synchronously refresh (block once, then serve fast forever).

2. **New file `pkg/agent/zehn_li_state_refresher.go`** (Zehn-only):
   - Background goroutine started from `wireZehnGitHubArtifactWriter`-style hook in `agent_init.go` (additive call only).
   - Runs the scan in-process with proper context cancellation and rate limiting.
   - Writes the workspace JSON file too (Phase 1 readers stay compatible).

3. **Prompt updates** point agents at `get_li_state` tool instead of reading the file.

Advantages over Phase 1:
- In-memory cache reads are sub-millisecond instead of file-I/O latency (negligible difference at our scale but cleaner).
- Type-safe Go structures instead of free-form JSON parsing in scripts.
- Refresh is driven by Go ticker, not a cron entry, removing one moving part.
- Naturally extensible to in-process webhook receiver in Phase 3.

Constraints:
- Requires a build + restart cycle.
- The Go code must be added as Zehn-only files (the upstream-clean discipline) so it doesn't conflict with future upstream sync. Companion plan `ZEHN_UPSTREAM_DAILY_SYNC_PLAN_20260516.md` should be in place before Phase 2 lands so the upstream-clean convention is enforced.

### Phase 3 — Webhook-driven invalidation

After Phase 2: wire GitHub webhooks to a public endpoint so cache invalidation happens on event, not on timer.

Components:

1. Public webhook receiver — Cloudflare Tunnel rule or similar fronting `:18791/webhook/github`.
2. Webhook signing key set per LI repo in `.picoclaw/.security.yml`.
3. Receiver routes events:
   - `issues.labeled` with `zehn:ready` → invalidate that repo's issue slice + republish `state_changed` event on the bus
   - `pull_request.opened` / `pull_request.synchronize` / `closed` → invalidate PR slice
   - `issues.closed` / `pull_request.merged` → invalidate corresponding slice + write terminal-outcome Yaad entry
4. Cache becomes near-real-time; the 10-min refresher becomes the backstop, not the primary path.

Benefits:
- New `zehn:ready` issue → agent picks it up within seconds, not minutes.
- Closed delegations / merged PRs immediately reflected in scoreboard.
- Reduces refresher load to near-zero (only used for "missed event" reconciliation).

This is the long-term goal but explicitly **not** required for the immediate efficiency win. Phase 1 alone solves ~95% of the problem.

## Migration / rollout

1. **Land the change in a PR** with the script and cache contract only. Do not commit runtime `.picoclaw` state or local cron entries.
2. **Ali approves merge.** Files land on main.
3. **Activate refresher locally**: prefer a LaunchAgent/supervised script while `tools.cron.allow_command=false`; use PicoClaw command cron only if command jobs are deliberately enabled, command confirmation is understood, and the gateway restart path is verified.
4. **Watch one refresh cycle**: observe cache file populated, `li_state_cache.status: ok`, freshness within tolerance.
5. **Update one consumer prompt at a time** (start with `zehn-operations-monitor.md` since it's read-only; lowest risk).
6. **Observe** for 24h. The ops monitor should report turn durations dropping dramatically while still flagging the same actionable issues.
7. **Roll out to heartbeat work-selection** (highest-cost consumer). Validate that no claimable work is missed by comparing the next 48 heartbeats' decisions against what the old inline scan would have decided.
8. **Roll out to CEO daily sync**. Should now feel snappy: 30-60s instead of multi-minute.
9. **Document the win** in supervision and update `LOGICIGNITER_OPERATING_CADENCE.md` to formalize cache-first behavior.

## Risks & mitigations

| Risk | Likelihood | Mitigation |
|---|---|---|
| Refresher cron silently broken; cache goes stale | Medium | Ops monitor health check on `cache_meta.generated_at` age; fallback to scan on stale. |
| Two refreshers running concurrently (e.g., user manually invokes) | Low | Lockfile in `.picoclaw/workspace/state/li_state_cache.lock`; second instance refuses to start. |
| Cache schema evolves; old consumers can't parse | Medium | `cache_meta.schema_version`; consumers warn + fall back when version mismatch. |
| New issue lands in 10-min window; agent doesn't see it | Medium | Acceptable: existing 30-min heartbeat cadence has the same problem with worse latency. Phase 3 webhook receiver eliminates this. |
| Agents continue running the old inline scan despite prompt update | Medium | Add explicit "Source-of-Truth Hierarchy" to `LOGICIGNITER_WORK_SELECTION.md`: 1) cache, 2) Yaad, 3) live scan only on fallback. Reviewers reject delegations that bypass cache. |
| Refresher itself takes >10 min (worse than current) | Low | The scan logic is unchanged; the refresher just runs it ONCE per cycle instead of per-agent. Same wall-clock per scan; the win is the dedup. |
| Refresher gets stuck and never completes | Medium | Cron job-level timeout (`tools.cron.exec_timeout_minutes: 12`); refresher restart on stuck condition. |

## Open questions for Ali

1. **Refresher cadence**: 10 minutes proposed. Alternatives: 5 min (fresher, more cost), 15 min (cheaper, more staleness). Phase 3 webhooks eliminate this tradeoff entirely.
2. **Refresher owner**: should Phase 1 run as a LaunchAgent/supervised local script, or as an explicitly enabled PicoClaw command cron? Recommendation: LaunchAgent/supervised script first, because command cron is currently disabled and this job is mechanical state collection, not agent reasoning.
3. **Phase 2 timing**: should we wait for the upstream sync to land before adding Go-side caching code, so the new Go files inherit the post-sync `agent_init.go`? Recommend yes.
4. **Cache scope creep**: this plan covers issue/PR/repo state. Should it also include the Release Ladder snapshot (`LADDER_SNAPSHOT_LATEST.md`)? They're conceptually similar caches. Suggest yes in Phase 1 or 2.
5. **Should `LOGICIGNITER_WORK_SELECTION.md` be updated to formally declare the cache as the canonical source**, deprecating the inline scan in doctrine? Recommend yes once Phase 1 is proven.
6. **Token cost & rate-limit estimate**: the refresher hits `gh` API at ~200 calls per cycle × 144 cycles/day = ~28,800 calls/day. GitHub's REST API limit for authenticated users is 5,000 req/hour = 120,000/day, so we're well under. But worth verifying actual usage on the existing setup before activating.

## Out of scope (intentional)

- Replacing the LLM-decision part of work-selection. The LLM still ranks and chooses from filtered claimable items per `LOGICIGNITER_WORK_SELECTION.md` — that's working correctly and is the right place for judgment. We're only fixing the *input* step.
- Caching Yaad reads. Yaad is fast enough already; reads are direct MCP calls returning small structured payloads.
- The "company-state-as-Yaad-entries" approach. Tried-and-rejected pattern above; Yaad isn't the right store for fast-changing operational data.
- Webhook implementation in this plan. Phase 3 only; needs its own design pass.
