# COO Daily Scoreboard Schema

Status: canonical schema for the daily scoreboard file generated by
`li-coo`. Owner: `li-coo`.

Purpose: give the company a single instrument. Without a scoreboard,
operating is invisible and "is anything working?" is unanswerable.

## File Layout

The COO writes two files per scoreboard run:

- `workspace/memory/scoreboard/YYYYMMDD.md` — the dated daily entry.
- `workspace/memory/scoreboard/LATEST.md` — overwritten copy of the
  latest daily entry, so other agents have a single canonical pointer.

## Required Sections

Every daily scoreboard MUST contain these sections, in this order, with
this exact heading text so the Operations Monitor can parse them.

### `## Header`

- Date (ISO-8601).
- COO agent ID (always `li-coo`).
- Generation time (ISO-8601).
- Yaad reachability at generation time (`reachable` | `degraded`).
- Verify-pr.sh wrapper version available in `logicigniter/scripts`.

### `## Weekly Goal`

- The current week's CEO-stated outcome.
- "On track" / "At risk" / "Slipping" status with one-line rationale.
- Days remaining in week.

### `## Release Readiness Ladder Aggregate`

Counts from `LOGICIGNITER_RELEASE_READINESS_LADDER.md`:

| Stage | Today | Yesterday | Δ |
| --- | --- | --- | --- |
| 1 Skeleton | n | n | ±n |
| 2 Backend | n | n | ±n |
| 3 Frontend | n | n | ±n |
| 4 Integration | n | n | ±n |
| 5 Quality | n | n | ±n |
| 6 Docs | n | n | ±n |
| 7 Launch-Ready | n | n | ±n |
| assessment-pending | n | n | ±n |

Per-suite progress (closest stage per suite, since the suite gates on
its lowest-staged app):

| Suite | Lowest Stage | Lowest Service | Highest Stage Reached |
| --- | --- | --- | --- |

### `## Yesterday's Terminal Outcomes`

Counts from Yaad reads of `organization:logicigniter` entries created
yesterday with class `decision`:

- Merged: count + links.
- Reviewed-and-approved: count + links.
- Blocked-with-owner: count + links.
- Escalated-to-Ali: count + links.
- Delegated-with-evidence-expectation: count + links.
- Deferred-with-retry-date: count + links.
- Replaced-or-closed: count + links.

A zero-count row across all categories is a red flag — that means the
prior day produced no durable record. The COO must investigate and
report why.

### `## Stuck-Work Register`

Items with no state change in >72h. Includes:

| Repo | Issue/PR | Age | Last State | Owner | Next Action | Retry Date |
| --- | --- | --- | --- | --- | --- | --- |

### `## Open PR Review Queue`

| Repo | PR | Area | Author | Opened | Reviewer | Last Reviewer Activity |
| --- | --- | --- | --- | --- | --- | --- |

PRs where last reviewer activity >24h ago are flagged.

### `## Today's Top 3 Specialist Actions`

The COO names three concrete items the sweep at 08:00 should delegate.
Format:

1. `<specialist-id>` — `<repo>#<issue>` — terminal target.
2. `<specialist-id>` — `<repo>#<issue>` — terminal target.
3. `<specialist-id>` — `<repo>#<issue>` — terminal target.

Fewer than 3 is acceptable only when the queue genuinely has fewer
claimable items; in that case the no-work-found report must accompany.

### `## Failure Register (Last 24h)`

Every non-terminal response containing a `Failure Reason:` line from
the last 24h, deduplicated:

| Time | Agent | Tool/Action | Failure Reason | Recurrence (within 7d) |
| --- | --- | --- | --- | --- |

Recurrence ≥3 promotes the failure to an operating-model issue.

### `## Yaad Activity (Last 24h)`

- Read calls (count).
- Write calls (count, by class).
- Failed Yaad calls (count + reasons).
- Entry IDs from yesterday's terminal outcomes.

If material terminal outcomes occurred but Yaad write/update count is zero on a
non-weekend day, the COO must flag the loop as broken. If no material outcomes
occurred, zero writes is acceptable and helps avoid duplicate no-work or
unchanged-blocker clutter.

### `## Verify-Pr Activity (Last 24h)`

- Total runs (by repo).
- pass / fail / skipped counts.
- Average duration.
- Repos missing the workflow (target: all consumer repos, currently
  only `scripts` has it; rollout via Artifact 2 composite action).

### `## Approval Queue`

Items labeled `approval:ali-required` waiting for Ali:

| Repo | Issue/PR | Asking Role | Question | Age |
| --- | --- | --- | --- | --- |

Items >7 days are flagged as stale-approval candidates.

### `## Anti-Patterns Observed Today`

Anything an agent did that violated:
- `LOGICIGNITER_ENGINEERING_QUALITY_DOCTRINE.md`
- `LOGICIGNITER_COMPANY_OPERATING_CONTRACT.md`
- This cadence doc.
- `LOGICIGNITER_YAAD_SCHEMA_CONTRACT.md`.

Patterns observed should be written to Yaad as class `anti_pattern`
with a remediation note.

## Generation Rules

- The scoreboard is generated daily at 08:30, before the 08:00
  specialist sweep's effects from that same day. The previous day's
  data is the input.
- If a section has no data (e.g., zero open PRs), include the heading
  and write `_none today_` so downstream parsers can rely on heading
  presence.
- Total file size should be < 500 lines. If it grows beyond that,
  truncate the registers to top-N and add `… N more, see Yaad query
  <q>` lines.

## Read Pattern

Agents that need company state should read `LATEST.md` first, not
recompute aggregates from raw data. The scoreboard is the cache; Yaad
+ GitHub are the sources of truth.

## Acceptance For First Drill

When the drill issue (Artifact 12) is run, the scoreboard for that day
must show:
- The drill issue in either "Today's Top 3 Specialist Actions" or
  "Stuck-Work Register" depending on its state.
- The verify-pr.sh evidence count incremented by ≥1.
- A Yaad write/update logged for the material terminal outcome, or an explicit
  reference to the equivalent active Yaad entry if it already existed.
- Failure register either empty (success) or containing the precise
  failure reason (clean failure).
